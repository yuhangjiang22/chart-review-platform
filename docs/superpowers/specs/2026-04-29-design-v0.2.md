| Area | v0.1 | v0.2 | Why |
|---|---|---|---|
| Field shape | `id`, `prompt`, `answer_schema`, `cardinality`, `time_window`, `derivation`, `is_final_output`, `extraction_guidance`, `group`, `guidance_prose` | + `is_applicable_when` | NCCN/HEDIS/quality-measure tasks routinely require gates ("only evaluate field X when field Y is true"). ## 2. New contract details

### 2.1 `is_applicable_when` (Field-level)

Optional string expression in the same dialect as `derivation`. Evaluated against the current leaf-value environment.

- **True** → field is applicable; agent answers it normally.
- **False** → field auto-resolves to the answer literal `'not_applicable'`. Agent skips it. UI greys the field, displays an N/A pill plus an explanatory `NotApplicableView` body that names the gate. The value `'not_applicable'` propagates through downstream derivations: any derivation that references a not-applicable input itself short-circuits to `'not_applicable'`.
- **References-an-input-not-yet-set** → status is `'unknown'`; treated as pending (not gated). Resolves once the input is filled.

Implementation: `chart_review.derivation.compute_applicability(compiled_task, values) -> {field_id: 'applicable'|'not_applicable'|'unknown'}`.

Demo: `cytology_supports_lung_primary` is gated on `pathology_report_present == 'no'`. In `patient_demo` (pathology present), the field auto-resolves to N/A.

### 2.2 `contradicting_evidence` (CriterionAssessment-level)

Optional array of `{evidence: EvidenceTriple, reason_not_decisive: string}`.

Distinct from:
- **`coverage.excluded`** — notes filtered out *before* reading (e.g., wrong author role, outside window, document type mismatch). The agent never read these.
- **`alternatives_considered`** — answer *values* the agent considered and rejected (e.g., "considered `nsclc`, rejected because the histology is small-cell").

`contradicting_evidence` lives between those two: text the agent retrieved AND read, that pulls toward a different answer, but that the agent ultimately weighed against. The reviewer sees it as a peer to the primary evidence card.

**Faithfulness applies equally.** `chart_review.faithfulness.check_assessment` runs the offset-vs-quote check on every contradicting-evidence row. An agent cannot cite contradicting evidence it cannot quote verbatim.

Demo: `oncologist_lung_cancer_diagnosis_in_note` returns `'yes'` and cites the oncology note, while listing the earlier pulm-note line "no diagnosis of lung cancer has been established" as contradicting evidence with reason "later oncology note + tissue confirmation supersede the earlier hedged note."

## 3. Operational additions

### 3.1 Synthetic patient corpus

Layout: `corpus/patients/<patient_id>/{meta.json, ground_truth.json, notes/<YYYY-MM-DD>__<doctype>.txt, omop/<table>.json}`.

Three JSON Schemas (`corpus/schemas/{meta,ground_truth,index}.schema.json`) constrain the per-patient files; `corpus/index.json` is a generated manifest of all patients with category + difficulty + headline.

- Hand-crafted (5): `patient_neg_hard_01`, `patient_probable_fhx_01`, `patient_confirmed_reread_01`, `patient_probable_cytology_01`, `patient_icd_z85_coexist_01`. Each is designed to fail a specific reasoning shortcut.
- API-generated (15): one per category cluster (NSCLC easy, SCLC easy, probable easy, ICD easy, negative easy). Generated via `tools/generate_corpus.py` against the Anthropic SDK; when no API key is configured, generation is delegated to Claude Code subagents.

Filesystem-as-API: agents work with bash primitives (grep, cat, jq) directly. The `chart_review.corpus` module just wraps those for platform code (CLI, batch runner, validation tests).

Helpers: `iter_patients`, `load_meta`, `load_ground_truth`, `read_note`, `grep_notes`, `omop_query`.

### 3.2 Async batch runner

`chart-review batch <compiled_task.json> [--all|--patients|--patients-file] --agent module:fn [--validate] [--concurrency N] [--run-id NAME]`.

Behavior:
- One thread per patient via `ThreadPoolExecutor` (`--concurrency` = max workers; default 5).
- Output to `runs/<run_id>/<patient_id>.json` plus `runs/<run_id>/summary.json`.
- `--validate` runs each ReviewRecord through `validate_review_record` before write; failed records become `summary.patients[].status = 'error'`.
- `runs/` is gitignored.

Agent contract: `run(compiled_task, corpus_root, patient_id) -> dict`. The compiled_task is shared by reference across workers — agents must treat it as read-only.

### 3.3 Reference agents and skills

**`chart_review.agents.stub:run`** — minimum-valid `no_info` for every leaf. Smoke-test target and benchmark floor.

**`chart_review.agents.lab_extractor:run`** — first non-stub. Iterates leaf fields; for fields registered in `_LAB_REGISTRY` (currently `lowest_hemoglobin_in_window`) invokes the extract_lab_value skill; otherwise falls through to `no_info`. Domain-honest: the agent never claims fields outside its registry.

**`chart_review.skills.extract_lab_value.extract`** — canonical skill shape:
1. **OMOP path** (deterministic, free, exact-date). Filter measurements by concept-name match + window; pick lowest value; normalize unit if recognized; refuse if unit unknown.
2. **Note path** (LLM, fallback). Grep notes for the lab's aliases, filter to in-window dates, send candidates to Claude with prompt caching on the system block, JSON-only output. Verify model-claimed offsets resolve to the exact quote in the source note before accepting; if not, refuse the value.
3. **No-API-key path**: returns `confidence='low'`, `missingness_reason='not_assessed'` rather than fabricating.

Anthropic client is **injectable** so unit tests cover the LLM path with a fake client (no API key needed in CI).

Coverage on the bundled corpus: 4/20 patients have a hemoglobin value the OMOP path extracts; the rest cleanly resolve to `null` answers. All 20 records validate against `review_record.schema.json`.

## 4. Reference task document

`tasks/lung_cancer_phenotype.md` now has 11 fields:

- 7 leaves: `pathology_report_present`, `pathology_lung_primary` (gated on report present), `cytology_supports_lung_primary` (gated on report absent), `imaging_lung_lesion`, `oncologist_lung_cancer_diagnosis_in_note`, `lowest_hemoglobin_in_window` (numeric/null, lab-value field), `icd_lung_cancer_present`.
- 4 derived: `pathology_confirms_lung_cancer`, `clinical_diagnosis_lung_cancer`, `pre_treatment_anemia_present` (= `Hgb != null AND Hgb < 12.0`), `lung_cancer_status` (final output).

The new lab field is the worked target of the lab_extractor reference agent. The new derived field demonstrates that the dialect handles `null` literals via short-circuit AND.

## 5. Decisions deliberately deferred (still open)

These came up during Phase 1 and were not closed. Phase 2 or later will revisit:

- `agent_should_answer: false` (or `field_type: human_judgment`) for holistic clinical judgment fields.
- Per-event-relative time windows (anchor on another field's value, not just `index_anchor`).
- `segmentation_rule` at task level for episode-based review units.
- Date precision + derivation flag on date answers/evidence.
- Episode `predecessor_relation` for LOT-style timelines.
- `state_trajectory` field type (smoking, alcohol, SDoH state).
- `review_unit: patient_year` formal support.
- `measure_specification_version` frontmatter convention.
- `is_screen_failure_reason` flag (UI nicety).

The above were surfaced in `docs/methodology/example-walkthroughs-and-required-functions.md` and `docs/methodology/chart-review-goals-and-design-fit.md`. None block Phase 1 exit.

## 6. What v0.2 does NOT change

- Three-layer trace exposure (primary visible, secondary on-click, tertiary forensic) — same.
- Validation-first UI principle — chart access is the primary surface, not agent reasoning.
- Three reviewer modes (quick verify / investigate / blinded) — same.
- Lifecycle states (`draft → piloting → revision_proposed → draft → locked`) — still aspirational; Phase 3 work.
- Migration policies (forward-only / targeted re-run / full re-run) — Phase 4 work.
- The four-tool data interface (`omop-query`, `concept-resolve`, `note-search`, `note-read`) — same. The corpus module wraps the synthetic-data adapter; real-EHR adapters slot in as Phase 4+.
