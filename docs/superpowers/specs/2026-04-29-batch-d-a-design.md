## 1 — Goal

Ship the multi-reviewer queue and stratified sampling features so a lead reviewer can:

1. **Sample** N patients stratified by configurable variables (age bucket, site, condition, etc.).
2. **Assign** sampled patients to reviewers (1-to-N — multiple reviewers per patient enables blinded calibration).
3. **Distribute work** via a per-reviewer queue (each reviewer sees only their assigned records).
4. **Track progress** — assignment status visible in QA panel rollup.

This closes Beat 5 (currently ◐ → ✓) and provides the substrate for any future multi-reviewer adjudication flows.

**Effort**: ~7-9 days. Schema (~0.5d) + sampling (~1.5d) + assignment server (~2d) + reviewer queue UI (~1.5d) + lead-reviewer Studio panel (~1.5d) + PatientList filter (~0.5d) + smoke/STATE (~0.5d).

**Beats moved**: Beat 5 ◐ → ✓.

**Out of scope** (Batch D-B or later): protocol version graph + migration UI; full multi-reviewer disagreement adjudication queue (with side-by-side blinded compare); reviewer skill/expertise routing; auto-load-balancing; stratified sampling QA dashboard.

## 4 — Stratified sampling

### 4.1 Algorithm

`app/server/sampling.ts` exports:

```ts
export interface SamplingInput {
  taskId: string;
  reviewsRoot: string;
  patientCorpusRoot: string;  // typically <PLATFORM_ROOT>/corpus
  sampleSize: number;
  stratifyBy: string[];       // empty = uniform random
  seed?: number;              // for reproducibility
}

export interface StratumGroup {
  key: Record<string, unknown>;  // e.g., { age_bucket: "65+", site: "Mayo" }
  patient_ids: string[];
}

export interface SamplingResult {
  total_eligible: number;
  strata: StratumGroup[];
  sampled: string[];           // resulting patient IDs
}

export function stratifiedSample(input: SamplingInput): SamplingResult;
```

**Algorithm**:

1. List all `corpus/<patient_id>/structured.json` files (the patient corpus already exists from Phase 1).
2. For each patient, extract values for each `stratifyBy` key from `structured.json`. Skip patients missing any key.
3. Group patients by their stratum tuple `{ key1: v1, key2: v2, ... }`.
4. Compute per-stratum sample count: `floor(sampleSize * stratum_size / total)`. Round up to ensure sum >= sampleSize. Cap at stratum_size.
5. Use seeded random (Mulberry32 or similar) to pick N from each stratum.
6. Return all sampled patient IDs + the strata structure.

If `stratifyBy` is empty, single stratum with all patients; sample uniform-random.

### 4.2 Endpoint

`POST /api/sampling/:taskId` — body `{ sample_size: number, stratify_by?: string[], seed?: number }`. Returns:

```ts
{ ok: true, total_eligible: 1850, strata: [...], sampled: ["p001", "p042", ...] }
```

Reviewer auth required. The endpoint just *computes* the sample; assignment is a separate step.

### 4.3 Tests

`app/server/__tests__/sampling.test.ts`:
- Build fixture: 100 patients, 60 with `age_bucket: "<65"`, 40 with `age_bucket: "65+"`. Sample N=20 stratified by age_bucket → expect ~12 from "<65" + ~8 from "65+" (proportional).
- Empty `stratifyBy` → uniform random.
- Sample size > stratum size → returns all stratum members + warns (via SamplingResult shape).
- Same seed → same sample (reproducibility).

## 5 — Assignment

### 5.1 Server module

`app/server/assignment.ts` exports:

```ts
export interface AssignmentInput {
  taskId: string;
  patientIds: string[];
  reviewerIds: string[];     // all reviewerIds get assigned to all patientIds
  reviewsRoot: string;
}

/** Mutate each (patient, task) review_state.json's assigned_to to include the reviewerIds.
 *  No-op if already assigned. Idempotent. Emits one record_assigned audit per assignment. */
export async function assignRecords(input: AssignmentInput, by: string): Promise<void>;

/** Remove reviewerIds from assigned_to. Emits record_unassigned audit per removal. */
export async function unassignRecords(input: AssignmentInput, by: string): Promise<void>;

/** Read the queue for a single reviewer: all (taskId, patientId) pairs assigned to them. */
export interface QueueEntry {
  task_id: string;
  patient_id: string;
  review_status: string;
  assigned_at: string;
}
export function getReviewerQueue(reviewerId: string, reviewsRoot: string): QueueEntry[];
```

### 5.2 Endpoints

- `POST /api/assignments/:taskId` — assign. Body: `{ patient_ids: string[], reviewer_ids: string[] }`. Reviewer auth.
- `DELETE /api/assignments/:taskId` — unassign. Body: same. Reviewer auth.
- `GET /api/queue/:reviewerId` — read queue. Reviewer auth (any reviewer can read any reviewer's queue, for cross-reviewer visibility — could tighten to "self only" if privacy concern).
- `GET /api/queue/me` — read own queue (sugar for the most common case).

### 5.3 Audit step types

```ts
| (BaseEntry & {
    step_type: "record_assigned";
    patient_id: string;
    reviewer_id: string;     // the reviewer being assigned
    assigned_by: string;     // the lead reviewer doing the assignment
  })
| (BaseEntry & {
    step_type: "record_unassigned";
    patient_id: string;
    reviewer_id: string;
    unassigned_by: string;
  })
```

### 5.4 Apply via applyUiAction or direct mutation?

Direct mutation through `mutate()` is the cleanest path — preserves the optimistic concurrency + broadcast pipeline. Add a new UiAction variant:

```ts
| { type: "set_assigned_to"; payload: { assigned_to: string[]; assigned_by: string } }
```

`applySetAssignedTo` updates `state.assigned_to`. The lock guard from Tier A already rejects writes when locked — assignment changes also rejected (good: locked records can't be re-assigned).

### 5.5 Tests

`app/server/__tests__/assignment.test.ts`:
- Assign alice + bob to p1 → review_state.json's `assigned_to` is `["alice", "bob"]`. Two `record_assigned` audit entries.
- Unassign bob → assigned_to is `["alice"]`. One `record_unassigned` audit.
- Idempotency: assign alice again → no-op, no new audit entry.
- Locked record rejects assignment with RECORD_LOCKED.

## 6 — Assignment-aware write guard

### 6.1 Behavior

When `assigned_to` is non-empty, only assigned reviewers can write field assessments. Non-assigned reviewer writes return 403 ASSIGNMENT_REQUIRED.

When `assigned_to` is empty (default), any reviewer can write — preserves current behavior. This makes the feature opt-in (assign records to flip-on enforcement).

### 6.2 Implementation

In `applyUiAction`'s `mutate()`, after the lock guard, before the actual mutation:

```ts
// Assignment guard — when assigned_to has entries, only assigned reviewers can write
// field assessments. Lead reviewers can still set_assigned_to / set_review_status.
if (
  current.assigned_to &&
  current.assigned_to.length > 0 &&
  by === "reviewer" &&
  action.type === "set_field_assessment" &&
  !current.assigned_to.includes(by_id)
) {
  throw new ReviewStateError("ASSIGNMENT_REQUIRED", `Reviewer "${by_id}" is not assigned to this record`);
}
```

### 6.3 Tests

Append to `assignment.test.ts`:
- Reviewer alice assigned, alice writes → succeeds.
- Reviewer bob NOT assigned, bob writes → ASSIGNMENT_REQUIRED.
- assigned_to empty, anyone writes → succeeds (back-compat).

## 7 — Client UI

### 7.1 PatientList filter pill

`PatientList.tsx` gains a small pill above the patient list: `[ all | mine ]`. When `mine`, filter out patients where `assigned_to` doesn't include current reviewer_id. Default: `mine` (assuming queue-driven workflow).

When `assigned_to` is empty (back-compat), pill toggle is no-op — show all.

State persisted to localStorage as `chartReview.queueFilter`.

### 7.2 New top-level "My Queue" view (optional / minimal v1)

For v1, the PatientList filter is sufficient. A separate "My Queue" view (with task-grouping, sort by status, etc.) would be richer but adds scope.

**Decision**: skip separate MyQueue.tsx in v1; the PatientList filter + existing layout serves the queue-driven workflow well enough. Leave `MyQueue.tsx` as a follow-up if needed.

### 7.3 Lead-reviewer AssignmentPanel (Studio extension)

`AssignmentPanel.tsx` — new tab in Studio (alongside AuthoringPanel + CohortPanel + MethodologistTokenPanel from Tier A):

Features:
- **Sampling**: select task, set sample size, optional stratification field selector (populated from `compiled_task.stratify_by`). Click "Sample" → POST to `/api/sampling/:taskId` → display results (count per stratum + total).
- **Assignment**: select reviewer(s) from the auth allowlist, select patients (from sampling result or manually-typed list), click "Assign". POST to `/api/assignments/:taskId`.
- **Existing assignments**: list of currently-assigned (patient × reviewer) pairs across all task records. Click row to unassign.

UI: 3 sub-sections in a single panel. Use existing atoms (Pill, Icon).

### 7.4 No CriterionPane changes

CriterionPane already hides action buttons when locked. The new ASSIGNMENT_REQUIRED guard manifests at write-time. Reviewer sees the error in toast/alert when they try to write to an unassigned record. Acceptable for v1; richer "you're not assigned" UI is a polish task.

## 8 — Strata data source

### 8.1 Where do `age_bucket`, `site`, etc. come from?

The corpus has `corpus/<patient_id>/structured.json` per patient (existing, OMOP-shaped). Stratification reads top-level fields from that JSON.

Convention: the keys named in `compiled_task.stratify_by` must exist as top-level keys in `structured.json`. If a patient lacks a key, they're excluded from sampling for that stratification (logged as `skipped` in the SamplingResult).

### 8.2 Pre-computed strata for performance

For tasks with many patients (1800+), repeatedly walking `corpus/<*>/structured.json` for every sampling call is wasteful. Cache the strata lookup:

`corpus/_strata/<task_id>.json` is generated lazily by the first `/api/sampling/:taskId` call. Subsequent calls read the cache. Manually invalidate by deleting the file (rare — strata don't change unless the corpus changes).

For v1, skip the cache. ~1800 file reads is sub-second; optimize later if needed.

## 10 — Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Reviewers without assignment can't write — surprises lead reviewer | medium | medium (workflow break) | Default `assigned_to` is empty (back-compat); lead reviewer must explicitly assign to flip on enforcement. Document in STATE.md + Studio panel |
| Stratification key missing in structured.json — patient excluded silently | medium | low (audit trail records `skipped`) | SamplingResult includes `skipped: { reason: "missing_key", patient_id: "..." }[]` so the lead reviewer sees the count |
| Sampling N > stratum size — produces fewer than N | high | low | Return `actual_sample_size` in result; UI shows discrepancy |
| Multi-reviewer assignment for blinded calibration writes — first reviewer's write blocks second's? | medium | medium | The existing `applyUiAction` is per-(patient×task), so two reviewers writing the same record race on the optimistic concurrency. In v1, the LATER write wins. v2 could add per-reviewer write namespacing for true blinded mode (see Phase B's BlindedReviewControls — that already supports per-criterion blinded; multi-reviewer at the record level would require extending it) |
| AssignmentPanel UI for selecting reviewers requires reviewer allowlist read access | low | low | Existing `auth.ts` exposes the allowlist via `process.env.REVIEWERS` — read in Studio |
