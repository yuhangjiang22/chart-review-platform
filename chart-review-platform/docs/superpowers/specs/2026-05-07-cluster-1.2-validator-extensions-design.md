# Cluster 1.2 — validator extensions

Three build-skill-layer gaps the cluster-1+1.1 validator cannot catch. All three share a single code surface (`lib/chart_review/build_skill_validator.py` + `contracts/criterion-file.schema.json`) and share fixture/test machinery.

## Scope

| # | Source | What the validator should additionally catch |
|---|---|---|
| **PS-2** | persona stress test | A derivation expression that does not match the documented threshold (e.g. `if total >= 8` when prose says `> 8`). |
| **PS-3** | persona stress test | A criterion whose prose says "at index" / "currently" but ships a `time_window`; or says "history of" / "ever" but ships no window. |
| **C1.1-OS-1** | cluster 1.1 explicit out-of-scope | Ref-check is currently hard-coded to two expression-shaped fields (`derivation.expr`, `is_applicable_when`); future fields need the same treatment. |

**Out of scope** (deferred):
- PS-1, PS-4, PS-5 — sequenced into clusters 1.4, platform-side, 1.3 respectively.
- C1.1-OS-2 (type compatibility) — needs `answer_schema` → expression-type integration; bigger project.
- C1.1-OS-3 (DSL parser) — explicit "only if DSL grows."

## Design

### PS-2 — derivation truth table

**Author writes a small, optional truth table on the derived criterion's frontmatter.** Each row is a leaf-input combination plus the expected derivation output. The validator evaluates the derivation against each row and asserts equality.

Schema addition (`contracts/criterion-file.schema.json`):

```json
"derivation_truth_table": {
  "type": "array",
  "minItems": 1,
  "items": {
    "type": "object",
    "additionalProperties": false,
    "required": ["inputs", "expected"],
    "properties": {
      "inputs":   { "type": "object" },
      "expected": {},
      "label":    { "type": "string" }
    }
  },
  "description": "Optional. Each row asserts that derivation(inputs) == expected. Authors are encouraged to include boundary rows (off-by-one thresholds) and any-null defaults."
}
```

Worked example (CHA₂DS₂-VASc total → 3-tier):

```yaml
derivation:
  kind: expression
  expr: |
    if total >= 2 then "high"
    else if total == 1 then "moderate"
    else "low"
derivation_truth_table:
  - { label: "below threshold", inputs: { total: 0 }, expected: "low" }
  - { label: "just below high", inputs: { total: 1 }, expected: "moderate" }
  - { label: "boundary",        inputs: { total: 2 }, expected: "high" }
  - { label: "above threshold", inputs: { total: 9 }, expected: "high" }
```

Validator behavior:

| Condition | Diagnostic | Level |
|---|---|---|
| Derived criterion has no `derivation_truth_table` | `derivation_no_truth_table` | warning |
| Row's `evaluate(expr, inputs) != expected` | `derivation_truth_table_mismatch` | error |
| Row's `inputs` references a `field_id` not declared in the package | `unknown_field_reference` (existing code, reused; message names source as `derivation_truth_table.inputs[<row_index>]`) | error |
| `evaluate()` raises | `derivation_eval_error` | error |

**Why optional and not required:** existing drafts (cha2ds2-vasc, rucam-score-v2, nccn-nsclc-adjuvant, plus the four persona drafts) ship without truth tables. Requiring them would retro-break clean drafts. The build skill's Phase 4.5 (derivation expressions) gains a sub-step that asks the user for boundary cases and emits the table. Once enough drafts have them, a future cluster can flip the schema to required.

### PS-3 — time-window discipline heuristic

**Regex over `prompt` + body for trigger phrases that suggest the criterion's temporal stance, cross-checked against the `time_window` field.** Warning-level: heuristics have false positives, so they surface in pre-flight rather than blocking validation.

Trigger sets (case-insensitive, word-boundaried):

| Stance | Phrases | Expected `time_window` |
|---|---|---|
| Point-in-time | `at index`, `currently`, `now`, `at the index date`, `today`, `this admission`, `at presentation` | absent |
| Windowed | `history of`, `ever`, `prior`, `past`, `any prior`, `lookback`, `previous` | present |

Diagnostic logic:

```
If criterion has time_window AND prompt+body contains any point-in-time phrase:
    emit time_window_likely_unneeded (warning)

If criterion has NO time_window AND prompt+body contains any windowed phrase:
    emit time_window_likely_missing (warning)
```

Each diagnostic message names the matched phrase verbatim so the author can decide quickly whether the heuristic is right (e.g. "matched 'history of' in body line 12 — should this criterion declare a time_window?").

**False-positive escape hatch:** authors can suppress on a per-criterion basis with frontmatter `time_window_check: skip` (default: not present, runs the check). Documented in `interview-guide.md` Phase 4.6 as "use this when the heuristic misfires (e.g. you literally have the phrase 'history of' inside an enum value)."

### C1.1-OS-1 — ref-check extensibility refactor

**Refactor only.** No schema growth. Today the validator hard-codes two expression-shaped fields:

```python
if isinstance(fm.get("is_applicable_when"), str):
    expressions_to_check.append((str(md), "is_applicable_when", fm["is_applicable_when"]))
deriv = fm.get("derivation") or {}
if isinstance(deriv, dict) and isinstance(deriv.get("expr"), str):
    expressions_to_check.append((str(md), "derivation.expr", deriv["expr"]))
```

Replace with a registry walked over a single list:

```python
EXPRESSION_FIELDS: list[ExpressionField] = [
    ExpressionField(name="is_applicable_when", extractor=lambda fm: fm.get("is_applicable_when")),
    ExpressionField(name="derivation.expr",    extractor=lambda fm: (fm.get("derivation") or {}).get("expr")),
]
```

Adding a future `prerequisite:` block (or any other expression-shaped field) becomes one line. No new diagnostics in this cluster.

### Cross-cutting — diagnostic levels

Current return shape:

```python
{"ok": True/False, "diagnostics": [{"code": ..., "path": ..., "message": ...}, ...]}
```

Any diagnostic flips `ok` to false.

New shape:

```python
{
  "ok": True/False,
  "diagnostics": [
    {"code": ..., "path": ..., "message": ..., "level": "error" | "warning"},
    ...
  ]
}
```

Rules:
- Default level is `"error"` (back-compat for existing diagnostic codes).
- `ok` is `False` iff at least one diagnostic has `level == "error"`.
- All existing codes (`missing_meta`, `meta_schema_violation`, `criterion_schema_violation`, `missing_frontmatter`, `todo_marker_in_body`, `unknown_field_reference`) remain errors.
- New warnings: `derivation_no_truth_table`, `time_window_likely_unneeded`, `time_window_likely_missing`.
- New errors: `derivation_truth_table_mismatch`, `derivation_eval_error`.

The MCP `validate_package` tool exposes the same shape; the Author pre-flight (cluster 6, future) renders warnings inline below errors.

## Build-skill prompt update

`interview-guide.md` Phase 4.5 (derivation expressions) gains a sub-step:

> After writing the derivation expression, propose 3–4 boundary truth-table rows: at-threshold, just-below, just-above, and an any-`null` row. Ask the user to confirm the expected values, then emit a `derivation_truth_table` array on the criterion frontmatter. If the user declines, leave it off — the validator will warn but not fail.

Phase 4.6 (per-criterion time windows) gains:

> The validator runs a heuristic on this criterion's prose: "at index"-style phrases without a time_window, or "history of"-style phrases with one, surface as warnings. If the heuristic misfires for a legitimate reason (e.g. an enum value literally contains the phrase), set `time_window_check: skip` on the frontmatter and document why in the criterion body's "Boundary / failure modes" section.

## Acceptance criteria

1. All new tests are demonstrably red before the validator changes and green after.
2. The seven existing real-world drafts still validate as `ok: True` (no error-level diagnostic regressions).
3. A new known-bad fixture per check exists and is exercised by tests.
4. The MCP `validate_package` tool returns the new shape (`level` field on every diagnostic).
5. The build skill, when run on a fresh task, emits a `derivation_truth_table` for at least one derived criterion (verified manually in a fresh-session smoke run).
6. `interview-guide.md` Phase 4.5 + 4.6 are updated and committed.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| `evaluate()` semantics differ between validate-time and runtime (e.g. truthiness of `null` operands) | Truth tables run through the same `evaluate()` function the runtime uses; no parallel implementation. |
| Time-window heuristic produces too many false positives on existing drafts | Warning-level + `time_window_check: skip` escape hatch; regression sweep on existing drafts gates the rollout. |
| Authors skip the optional truth table → no actual coverage gain | The build-skill prompt change makes it the default; warning surfaces missing tables; future cluster can require it. |
| Refactoring the ref-walker breaks existing diagnostics | Cluster-1.1 typo fixture stays in the test suite as a regression gate. |

## Out of scope (next-cluster work)

- **Required truth tables** (after enough drafts have them).
- **Truth-table coverage analysis** (e.g. "did the table cover all enum values of each input?"). Useful but adds combinatorial logic; not needed for v0.
- **Time-window severity grading** (some "at index"/time_window mismatches are harmless paraphrases). The current binary warning is sufficient until evidence shows the heuristic is too noisy.
- **`prerequisite` schema field** (C1.1-OS-1's forward-compat counterpart). Refactor only here; add the field when it has a real consumer.
