## What the new ref-check buys

The `unknown_field_reference` diagnostic catches:

- Bare typos in `derivation.expr` (`chf_presnt` instead of `chf_present`)
- References to fields that were renamed but the derivation wasn't updated
- References to fields the author intended to add but didn't (latent bug — see fixture story below)
- Same checks on `is_applicable_when` (e.g. a gate referencing a non-existent `eligibility_check`)

## Latent bug surfaced by the ref-check

Implementing the ref-check exposed a pre-existing inconsistency in the `known-good` fixture: `derived_status.md` referenced two leaves (`lung_cancer_imaging_suspicious`, `lung_cancer_clinical_mention`) that didn't exist as criterion files in the same package. The fixture validated clean under the cluster-1 schema (which doesn't cross-reference) but would have failed at runtime when the contract evaluator tried to resolve those leaves.

The fix added the two missing leaf criterion files. This is exactly the kind of authoring bug the ref-check is designed to catch, and validates the approach: the typo-or-rename gap is real, latent, and silent until runtime — and now it's loud and at the validation step.

## Out of scope (future work)

- **Reference check on selected_evidence.field_id and prerequisite chains.** The current pass only walks `derivation.expr` and `is_applicable_when`. If criterion frontmatter ever gains other expression-shaped fields (e.g. a `prerequisite:` block listing field_ids the criterion depends on), they need the same treatment.
- **Type-compatibility checks.** A derivation that does `chf_present + 1` would currently pass (chf_present is treated as a name reference); we don't verify the expression is semantically valid against each leaf's `answer_schema`. The contract evaluator does this at runtime; lifting it to validation time is a larger project.
- **DSL grammar parsing.** The current ref-extractor is a regex over identifiers, not a real parser. It cannot tell `if foo` (where `foo` is a name) from `if "foo"` (a quoted literal that happens to look identifier-like — though our quote-stripping handles that for now). For the scenarios cluster 1.1 targeted, the regex is sufficient; a proper parser is justified only if the DSL grows.
