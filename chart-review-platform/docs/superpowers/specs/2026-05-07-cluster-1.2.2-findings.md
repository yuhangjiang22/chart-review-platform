## Boundary coverage strategy

For each derivation, rows aim to exercise:

1. **Threshold-adjacent values** — at the threshold, just below, just above.
2. **Each branch of an if/else chain** — including the final `else` fallback.
3. **Applicability gates** — rows that hit the `not_applicable` short-circuit when present.
4. **Each leaf-input combination that produces a distinct output** — enough to lock the input → output mapping.

### Worked example — `dr_screening_concordance`

The derivation has 4 input fields and 3 outcomes, with 3 ANDed conditions in the concordant branch:

```
if is_diabetic == "no" then "not_applicable"
else if eye_exam_in_window == "yes" and eye_exam_was_dilated_retinal == "yes" and eye_exam_by_eye_care_pro == "yes" then "concordant"
else "discordant"
```

5 rows cover: gate (not diabetic), all-three-met (concordant), each AND-condition individually negated (3 × discordant). This exercises every distinct path through the boolean expression — a 4-row table would have left one of the three AND conditions un-isolated, masking a hypothetical typo if e.g. the derivation were rewritten with one of the conjuncts broken.

## How the validator caught wrong values during authoring

The truth-table runner runs each row through `chart_review.derivation.evaluate()` and emits `derivation_truth_table_mismatch` (error) when `actual != expected`. During this cluster the implementer's first-pass rows hit several mismatches — each one named the exact row and the actual evaluator output, allowing the author to either correct the `expected` value or revise the input combination. The feedback loop is tight enough that boundary authoring becomes mechanical once the derivation logic is understood.
