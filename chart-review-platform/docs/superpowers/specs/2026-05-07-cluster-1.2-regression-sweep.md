seven existing drafts that pre-date the truth-table requirement.
This sweep confirms all seven still validate as `ok: True` and tallies
the new warnings so the next cluster can target adoption.

## Notable findings

### CHA₂DS₂-VASc (`chart-review-cha2ds2-vasc`)

**`time_window_likely_missing` — `cha2ds2_vasc_score.md` (phrase: "prior")**
The word "prior" appears in the examples section ("prior stroke/TIA history") and the boundary notes, not in a field-level prose sentence describing what the criterion looks back over. The `cha2ds2_vasc_score` criterion is a derived rollup score — its time_window semantics are entirely determined by the leaf criteria. This is a **false positive**. Candidate fix: add `time_window_check: skip` to `cha2ds2_vasc_score.md`.

**`time_window_likely_unneeded` — `diabetes_present.md` (phrase: "at index")**
The phrase "at index" appears in the boundary notes ("CHA₂DS₂-VASc counts active DM at index"). The criterion does carry `time_window: lookback_lifetime` to capture chronic DM diagnosis regardless of when it was first documented — a deliberate design choice. The "at index" mention describes the evaluation semantics, not that the criterion is point-in-time. This is a **borderline false positive**: the lifetime lookback is correct for a chronic condition, but the wording is ambiguous. Candidate fix: either add `time_window_check: skip` or reword the boundary note to avoid "at index."

**`derivation_no_truth_table` — `stroke_risk_tier.md`**
Derived criterion; no truth table yet. Adoption deferred to next cluster.

### Sepsis-3 (`chart-review-sepsis-3-p1`)

**`time_window_likely_unneeded` — `infection_suspected_at_index.md` (phrase: "at index")**
The phrase "at index" appears in the `prompt` field ("Was infection suspected at index — cultures of any source ordered AND systemic antibiotic started within ±24h of ED triage?"). The criterion has `time_window: peri_index_24h`, which is entirely consistent — the ±24h window is the defining feature of the Sepsis-3 infection suspicion criterion. The heuristic fires because "at index" appears in the same criterion that also has a time_window, but the phrase and the window co-exist correctly: the window specifies the ±24h range, "at index" identifies the anchor event. This is a **false positive**. Candidate fix: add `time_window_check: skip`.

**`derivation_no_truth_table` — `sepsis_present.md`**
Derived criterion; no truth table yet. Adoption deferred to next cluster.

### Statin Adherence (`chart-review-statin-adherence-p2`)

**`time_window_likely_unneeded` — `proportion_days_covered_180d.md` (phrase: "at index")**
The criterion has `time_window: lookback_180d`, which is semantically correct — PDC is measured over the 180 days preceding the index date. "At index" appears in the `is_applicable_when` expression (`statin_active_at_index == "yes"`) and in the definition prose gating note ("Gated on `statin_active_at_index == "yes"`"). The time_window is both present and correct; the phrase "at index" refers to the eligibility gate, not the PDC measurement window. This is a **false positive**. Candidate fix: add `time_window_check: skip`.

**`time_window_likely_missing` — `statin_active_at_index.md` (phrase: "prior")**
The phrase "prior" appears in the definition prose: "a fill in the last 90 days... such that supply could plausibly cover the index date." The criterion explicitly notes "Point-in-time evaluation — no time_window in frontmatter (per Phase 4.6: index-time attribute)." The 90-day fill check is an internal look-back baked into the definition, not a separate time_window field. The heuristic fires on the word "prior," but the criterion's design is intentionally window-free at the frontmatter level. This is a **false positive** (the criterion has a deliberate note documenting the absence of `time_window`). Candidate fix: add `time_window_check: skip`.

**`derivation_no_truth_table` — `adherence_label.md`**
Derived criterion; no truth table yet. Adoption deferred to next cluster.

### Diabetic Retinopathy Screening (`chart-review-dr-screening-p3`)

**`time_window_likely_missing` — `dr_screening_concordance.md` (phrase: "past")**
"Past" appears in the definition: "dilated retinal exam by an eye care professional in the past 12 months." This is a **real authoring signal**: the derived rollup itself describes a 12-month lookback window, but the `time_window` field is absent from the frontmatter. However, `dr_screening_concordance` is a derived criterion (kind: expression); the window semantics should live on the leaf criterion `eye_exam_in_window`, which likely already carries a `time_window: lookback_12mo`. The derivation doc is re-stating the leaf's window for human clarity. This is a **borderline case** — the derived criterion should not carry a redundant `time_window` (the leaf owns it), but the prose is confusing. Candidate fix: remove "past 12 months" from the derived criterion's definition and replace with "within the lookback window defined by `eye_exam_in_window`", or add `time_window_check: skip`.

**`time_window_likely_missing` — `eye_exam_was_dilated_retinal.md` (phrase: "past")**
The prompt reads "Was the eye exam in the past 12 months a dilated retinal exam?" and the definition refers to "at least one eye exam in the lookback window." The criterion is gated (`is_applicable_when: 'eye_exam_in_window == "yes"'`) and asks about the *quality* of an exam that already passed the window filter. No independent time_window is needed here. This is a **false positive** — the window is enforced by the gate criterion, not this one. Candidate fix: add `time_window_check: skip`.

**`time_window_likely_missing` — `is_diabetic.md` (phrase: "history of")**
The phrase "history of" appears in the non-satisfying example: "Hx of gestational diabetes 5 years ago, no current DM." The criterion is explicitly a point-in-time evaluation ("no time_window in frontmatter") assessing current active DM status. The "history of" phrase appears only in an exclusion example. This is a **false positive**. Candidate fix: add `time_window_check: skip`.

**`derivation_no_truth_table` — `dr_screening_concordance.md`**
Derived criterion; no truth table yet. Adoption deferred to next cluster.

## Next-cluster work surfaced

- Adoption of `derivation_truth_table` across the seven existing drafts
  (each derived criterion needs 3–4 boundary rows). Estimate: 7
  criteria × ~4 rows each = ~28 truth-table rows to author.
- False-positive rate on the time-window heuristic from this sweep:
  9 false positives out of 10 total time-window warnings (90%). This
  is well above the ~25% threshold for tightening the heuristic.
  Recommended actions for the next cluster:
  1. Add `time_window_check: skip` to the 9 false-positive criteria
     identified above as a near-term fix.
  2. Consider tightening the `time_window_likely_missing` heuristic to
     exclude matches inside example/boundary-notes sections (i.e., only
     scan the Definition and Extraction guidance sections, not Examples
     or Boundary/failure-modes).
  3. Consider excluding "prior" matches when it appears as part of a
     field name reference (e.g., "`prior_hepatotoxicity_score`") rather
     than free-standing prose.
  4. The 2 genuine signals are: `dr_screening_concordance.md` (borderline
     — derived criterion re-stating a leaf's window) and
     `diabetes_present.md` (borderline — lifetime lookback co-existing
     with an "at index" evaluation note). Neither requires an immediate
     schema fix, but both are candidates for prose clarification.
